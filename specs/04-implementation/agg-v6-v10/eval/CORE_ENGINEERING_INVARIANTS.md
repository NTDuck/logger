# The Engineering Crucible: v6 to v10 Post-Mortem

## I. The Kafka Backpressure Paradigm (Tracks 2, 3, 4)

* **The Flaw We Escaped:** By implementing backpressure via `consumer.pause()` followed by an indefinite `tokio::time::sleep` or retry loop directly within the main actor loop, the application permanently blocks the async executor from calling `consumer.recv().await`. This starves `librdkafka` of application-level polling. Once Kafka's `max.poll.interval.ms` (default 5 minutes) is breached, the broker assumes the consumer has died and forcefully revokes its partitions. When the downstream I/O eventually recovers, pending offset commits fail with a `CommitFailedException` because partition ownership was lost, triggering a cascading death spiral of rebalances and duplicate processing.
* **The v10 Mechanical Invariant:** The consumer architecture MUST structurally decouple into two separate Tokio tasks connected by a bounded `mpsc` channel. The Fetcher Task continuously loops `consumer.recv().await` and pushes to the channel. The Processor Task reads from the channel and executes all I/O and retry loops. When the Processor blocks in an exponential backoff loop, the bounded channel fills up, causing the Fetcher to naturally block on its channel send via TCP backpressure. Pre-fetched messages remain safely inside `librdkafka`'s internal queues while the background thread autonomously maintains heartbeat leases without application-level starvation.

## II. Socket-Level Memory Defense (Track 1)

* **The Flaw We Escaped:** Deserializing HTTP payload bytes directly into an intermediate memory representation like `serde_json::Value` or an AST struct forces the entire payload structure into memory *before* depth or size rules can be enforced. An attacker can craft a deeply nested JSON payload or one with massive arrays. The deserializer will perform unbounded memory allocations and recurse deeply, causing stack-overflow crashes and memory-exhaustion Denial of Service (OOM) during the AST construction phase, fundamentally bypassing downstream application logic. Furthermore, lacking a socket-level timeout allows attackers to drip-feed payloads at 1 byte per second, causing a Slowloris connection pool exhaustion.
* **The v10 Mechanical Invariant:** The ingestion pipeline MUST explicitly use a low-level token pull-parser (e.g., `struson`) to iteratively evaluate the raw byte stream. `{` and `[` tokens must be counted directly from the stream, allowing the system to abort with HTTP 400 immediately if nesting depth exceeds 5 levels BEFORE any memory allocation or AST construction occurs. Additionally, a strict `tower::timeout::TimeoutLayer` MUST be applied at the socket stream-reading phase to sever connections exceeding physical time-to-first-byte limits, completely neutralizing Slowloris and allocation DoS vectors.

## III. Egress Decoupling & TCP Blocking (Track 6)

* **The Flaw We Escaped:** Wrapping `ws.send().await` in `tokio::time::timeout` is a fatal operation for the Tungstenite protocol. If the timeout triggers, the future is dropped mid-transmission, leaving the underlying WebSocket framing corrupted and permanently destroying the TCP stream. Furthermore, executing the network send directly within the synchronous broadcast fan-out loop causes TCP Head-of-Line blocking. A single lagging client on a degraded network will stall the loop, blocking egress for all healthy clients connected to the same instance.
* **The v10 Mechanical Invariant:** Every WebSocket session MUST be physically decoupled into a 3-Task Ingress/Processor/Egress separation using local bounded channels. The Processor Task reads from the global broadcast channel and executes a non-blocking `.try_send()` into a bounded `mpsc` egress channel (capacity 256). The Egress Task exclusively drains the `mpsc` channel and executes the async `sink.send().await`. If a client degrades, the egress channel fills, `.try_send()` returns `Full`, and the Processor simply drops the sender to gracefully sever the lagging client without corrupting frames or blocking the global broadcast fan-out.

## IV. Telemetry & State Consistency

* **The Flaw We Escaped:** Placing metric increments (like `logger_events_processed_total`) inside a `while` loop that performs exponential backoff against a failed downstream system conflates *delivery attempts* with *event throughput*. A 5-minute database outage generates thousands of phantom "processed error" counts per stalled message. This fundamental corruption hallucinates massive traffic spikes in Prometheus when the system is actually stalled. Furthermore, performing non-transactional state mutations before external HTTP calls causes redelivery blackholes if the offset commit is skipped upon HTTP failure.
* **The v10 Mechanical Invariant:** Telemetry increments MUST strictly adhere to the rule of Terminal Execution. Metrics are counted EXACTLY ONCE outside of all retry loops when a message reaches its final resolution (successful delivery, DLQ routing, or explicit drop). To prevent commit tearing and duplicate states, dual-write pipelines MUST utilize the `log_id` as the Kafka Message Key to guarantee absolute idempotency on retry, and consumer offsets are ONLY committed after the terminal I/O sink has returned success.
