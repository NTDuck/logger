# State Machine & Ownership Audit Report: v8 Architecture

## Evaluator Confidence & Material Unknowns
- **Confidence Level:** High.
- **Material Unknowns:** The exact internal configuration of `librdkafka` properties (`max.poll.interval.ms` vs `heartbeat.interval.ms`) applied in the adapters. If `max.poll.interval.ms` is exceptionally high, manual heartbeat maintenance via `recv()` polling becomes completely unnecessary, mitigating some risks but exposing others.

## 1. The Pre-Fetch Data Loss & Unbounded Memory Trap
- **Evidence Inspected:** `specs/04-implementation/v8/track-03-db-writer.md` and `specs/04-implementation/v8/track-04-ai-consumer.md` (Anti-Blocking Backpressure directives).
- **Current Behavior:** To prevent Kafka broker eviction while a downstream system (ClickHouse) is offline, the DAG mandates pausing partitions and entering a backoff loop. Inside this loop, it instructs the `tokio::select!` to poll `consumer.recv()` and "buffer or discard the result" to maintain the consumer heartbeat.
- **Correctness Risks & Edge Cases:** `rdkafka` pauses partitions asynchronously. The internal pre-fetch queue already contains messages when pause is called. If the actor *discards* these messages to keep the loop moving, it causes permanent data loss (when the downstream write eventually succeeds, committing the current batch's offset implicitly skips all discarded messages). If the actor *buffers* these messages in an auxiliary vector, it creates a hidden, unbounded memory leak during prolonged downstream outages, violating the explicitly stated physical constraints of strict memory bounds.
- **Structural Rethink:** Rely on `librdkafka`'s background heartbeat thread for session liveness. To prevent `max.poll.interval.ms` broker eviction without data loss, structurally decouple the Kafka network fetching boundary from the processing/retry boundary. Introducing an explicit, bounded asynchronous channel between a dedicated fetcher task and the processing task allows the fetcher to naturally block on channel capacity (mechanical backpressure) without needing to manually juggle pre-fetch draining.

## 2. The `watch::Receiver` Shutdown Deadlock
- **Evidence Inspected:** `specs/04-implementation/v8/track-03-db-writer.md` (Phase 3, Step 4 & Phase 4 Graceful Shutdown).
- **Current Behavior:** The main actor loop polls `shutdown_rx.changed()`. Upon triggering, the instructions mandate that the actor break the loop and execute the "Flush Subroutine" to write the final batch. If this flush fails, it enters a retry loop which *also* contains a `tokio::select!` polling `shutdown_rx.changed()`.
- **Correctness Risks & Edge Cases:** `tokio::sync::watch::Receiver::changed()` operates on state transitions. Once the shutdown signal is consumed in the main loop, the receiver marks the version as seen. When the Flush Subroutine enters the retry loop and awaits `changed()` again, it will block indefinitely waiting for a *new* state change that will never arrive. If ClickHouse is offline during a deployment rollout, the application will deadlock in the retry loop and refuse to shut down.
- **Structural Rethink:** Transition cancellation signaling from state-transition primitives (`watch::Receiver`) to latch-based primitives (`tokio_util::sync::CancellationToken`). Latches are idempotent; once triggered, any subsequent or nested awaits will resolve immediately, ensuring safe recursive polling across nested state machines.

## 3. The Telemetry Double-Counting Paradox
- **Evidence Inspected:** `specs/04-implementation/v8/track-03-db-writer.md` and `specs/04-implementation/v8/track-04-ai-consumer.md` (Phase 3, Step 4: Flush Subroutine / Write Sidecar Batch).
- **Current Behavior:** The actor loop is instructed to increment the `logger_events_processed_total{status="error"}` counter *before* entering the infinite retry loop. If the retry loop eventually succeeds, the flow proceeds to the success path, which increments the `logger_events_processed_total{status="success"}` counter for the same batch.
- **Correctness Risks & Edge Cases:** A transient network partition will cause the exact same batch to be counted twice (once as an error, once as a success). This destroys the mathematical invariants of the telemetry (where the sum of processed events should equal the exact number of messages ingested). Dashboards will report phantom spikes in log volume that do not exist in reality.
- **Structural Rethink:** Telemetry increments must be exclusively bound to the terminal resolution of a batch, not to the state transitions of the retry loop. Increment only when the batch is definitively committed to the downstream system or definitively routed to a Dead Letter Queue. Track retry attempts via independent duration histograms.

## 4. Dropped Sink Futures in WebSocket Broadcast
- **Evidence Inspected:** `specs/04-implementation/v8/track-06-websocket-server.md` (Phase 3, Step 3: session_loop).
- **Current Behavior:** The `ws.send().await` operation is wrapped in a `tokio::time::timeout`. If the timeout expires due to TCP backpressure from a slow client, the future is dropped, and the WebSocket connection is forcefully broken.
- **Correctness Risks & Edge Cases:** Dropping a `Sink` future mid-yield (such as `tungstenite`'s send future) corrupts the underlying framing state. In this specific DAG, because the loop immediately breaks and severs the connection, the corrupted state is safely destroyed. However, if any future refactoring attempts to keep the connection alive after a timeout, the corrupted frame state will cause protocol-level panics or silent data corruption.
- **Structural Rethink:** While mechanically safe through operational ordering, it relies on brittle implicit assumptions. A safer structural pattern is to explicitly decouple the network write into a dedicated egress task that can be dropped entirely without risking frame corruption in the shared session logic.
